# 主动发送集成说明

## 1. 背景

`frida/send/frida_active_send_rpc.js` 通过 Frida RPC 暴露 `sendtext(talker, content, type)`，让外部脚本能够主动让微信发出一条文本消息。该能力已经以 Java 反射的方式集成进 WeChatMonitor LSPosed 模块，不再依赖 PC 端 Frida 主控端。

## 2. 公开 API

包：`com.wechat.monitor.sender`

```java
public final class ActiveSender {
    public static boolean warmUp(MonitorContext ctx);
    public static boolean isReady();
    public static SendResult sendText(MonitorContext ctx, String talker, String content, int type);
}

public final class SendResult {
    public boolean isOk();
    public String  getMessage();
    public Throwable getError();
}
```

## 3. 调用流程

1. 模块装配时，`hooks.ActiveSenderHook.install(...)` 自动调用 `ActiveSender.warmUp(ctx)`，预先解析并缓存 `px0.r1.a`、`px0.q1.e`、`px0.q1.h`、`px0.q1.b` 四个 `Method` 句柄。
2. 业务代码调用 `ActiveSender.sendText(ctx, talker, content, 1)`。
3. `sendText` 做参数校验：talker / content 非空，type 必须为 `1`（与 Frida 脚本保持一致的安全护栏），就绪状态判断（必要时再次 warmUp）。
4. 校验通过后立即 `Handler(Looper.getMainLooper()).post(...)` 入队主线程异步执行，方法本身立即返回 `SendResult.ok("已入队主线程发送...")`。
5. 主线程异步执行：
   1. `r1A.invoke(null, talker)` 构造 q1 对象（R1.a 是静态方法）
   2. `q1E.invoke(q1, content)` 写入内容
   3. `q1H.invoke(q1, type)` 写入类型
   4. `q1B.invoke(q1)` 触发实际发送
6. 每一阶段都通过 `MonitorContext.logMessage("主动发送", ...)` 输出日志。
7. 上述反射调用同时触发 `SendMessageHook` 的拦截器，按 1500ms 指纹去重后再输出 `发送链路 / 发送构造 / 发送Scene / 发送网络 / 发送回调` 链路日志，作为发送是否真正落地的反向验证。

## 4. 安全护栏

| 约束 | 说明 |
|---|---|
| 仅 type=1 | 与 frida 脚本一致，避免误发媒体 / 卡片 / 红包等非文本消息 |
| 空值拦截 | talker / content 任一为空直接返回 `SendResult.fail` |
| 反射缓存懒加载 | warmUp 失败不阻断模块，sendText 时会重试一次 |
| 主线程异步 | 调用者线程不阻塞，避免在 BroadcastReceiver / Hook 回调内死锁 |
| 异常隔离 | 反射 `Method.invoke` 抛出的 `InvocationTargetException` 等被 catch 后只写日志，不向调用者抛出 |

## 5. 日志输出

日志类型固定为 `主动发送`，写入位置：

- logcat：`adb logcat -s WeChatMonitor`
- 文件：`/sdcard/WeChatMonitor/logs/wechat_monitor_yyyyMMdd.txt`（UTF-8 无 BOM）

典型一次发送的日志序列：

```
[主动发送] ActiveSender.sendText 入队 | talker=wxid_xxx | content=ping | type=1
[主动发送] ActiveSender R1.a 完成 | q1=px0.q1 | talker=wxid_xxx
[发送链路] px0.r1.a | talker=wxid_xxx              ← SendMessageHook 拦截到
[发送构造] px0.q1.e(content) | arg=ping | state... ← SendMessageHook 拦截到
[发送构造] px0.q1.h(type) | arg=1 | state...       ← SendMessageHook 拦截到
[主动发送] ActiveSender Q1.e/Q1.h 完成 | content=ping | type=1
[发送Scene] px0.r0.<init> | argc=5 | args=[wxid_xxx, ping, 1, ...]
[发送网络] px0.r0.doScene | type=522
[主动发送] ActiveSender Q1.b 已调用 | talker=wxid_xxx
[发送回调] px0.r0.onGYNetEnd | args=4
```

## 6. 触发方式

模块提供 **3 种**主动发送触发通道,优先级从高到低:

| 触发器 | 实现 | 响应延迟 | 适用场景 |
|---|---|---|---|
| **SendCommandReceiver**(主路径) | Hook 进程内动态注册 BroadcastReceiver | 即时 | SettingsActivity 按钮 / adb am broadcast |
| **CommandPoller**(兜底) | Hook 进程 HandlerThread 1 秒轮询 send.json | ≤ 1 秒 | adb 写文件(Android 10 以下) |
| **SendTriggerHook**(兜底) | Hook `Activity.onResume`,过滤 LauncherUI | 切回微信主界面才触发 | 上面两个都失败时 |

**为什么不只用文件**:Android 11+ scoped storage 让微信进程没权限读 `/sdcard/WeChatMonitor/send.json`,文件路线在新机型上会 `EACCES (Permission denied)`,所以广播 IPC 才是主路径。

### 6.1 SettingsActivity 按钮触发(推荐)

1. 打开 WeChatMonitor App 设置页 → 「主动发送」区块。
2. 填 talker(默认 `filehelper`)和消息内容。
3. 点「立即发送」按钮。
4. SettingsActivity 进程构造 Intent,带 extras `talker / content / type`,显式 `setPackage("com.tencent.mm")` + `FLAG_INCLUDE_STOPPED_PACKAGES`,`sendBroadcast`。
5. 微信进程内的 `SendCommandReceiver` 立即收到,调用 `ActiveSender.sendText`。
6. 看 logcat:`[主动发送] 收到广播 action=...` 后跟 `ActiveSender ...` 序列。

### 6.2 adb 触发

```bash
# 方式 A:直接发广播(推荐,绕开文件权限)
adb shell am broadcast \
  -p com.tencent.mm \
  -a com.wechat.monitor.ACTION_SEND_TEXT \
  --es talker filehelper \
  --es content "ping" \
  --ei type 1

# 方式 B:写文件(仅 Android 10 及以下可用,Android 11+ 会 EACCES)
adb shell "echo '{\"talker\":\"filehelper\",\"content\":\"ping\",\"type\":1}' > /sdcard/WeChatMonitor/send.json"

# 看日志
adb logcat -s WeChatMonitor:* | grep 主动发送
```

### 6.3 命令文件格式(给 CommandPoller / 老 adb 脚本)

- 路径:`/sdcard/WeChatMonitor/send.json`(UTF-8)
- 格式:`{"talker":"wxid_xxx","content":"hello","type":1}`
- 字段:同 6.1,type 默认 1(仅文本)

### 6.4 防抖与去重

- `SendCommandFile.pollAndSend` 200ms 防抖,避免 CommandPoller 多次 tick 重复读盘
- 真正"同一命令不会发送两次":文件处理完就 rename 为 `.done.<ts>` 或 `.failed.<ts>`,下一轮 tick 找不到 `send.json` 直接跳过
- 广播路径没有去重,每次 sendBroadcast 都会触发一次发送 —— 调用方自行控制频率

### 6.5 失败处理

| 场景 | 结果 | 日志 |
|---|---|---|
| 广播 talker/content 缺失 | 不发送 | `talker/content 不能为空` |
| 广播 type ≠ 1 | 不发送 | `仅支持文本消息 type=1` |
| ActiveSender 未就绪 | 不发送 | `ActiveSender 未就绪` |
| send.json 不存在(文件路径) | 无 | `命令文件不存在` |
| send.json JSON 解析失败 | rename `.failed.<ts>` | `SendCommandFile JSON 解析失败` |
| 反射调用异常 | rename `.done.<ts>` | `ActiveSender 反射调用异常` |

> 反射调用阶段的异常被 catch 后只写日志,命令文件仍重命名为 `.done`,避免下次重复触发同一条已损坏的命令。

## 7. 临时验证(不通过广播/文件)

如果不想走广播或文件触发,也可以在某个已有 Hook 回调里硬编码调用 `ActiveSender.sendText`:

```java
ActiveSender.sendText(monitorContext, "filehelper", "ping " + System.currentTimeMillis(), 1);
```

注意:仅用于自检,验证完成后必须回滚。

## 8. 与 Frida 脚本的对应

| Frida 脚本符号 | LSPosed 等价 |
|---|---|
| `rpc.exports.sendtext(talker, content, type)` | `ActiveSender.sendText(ctx, talker, content, type)` |
| `rpc.exports.ping()` | `ActiveSender.isReady()` |
| `Java.use('px0.r1').a(talker)` | `r1A.invoke(null, talker)` |
| `q1.e(content)` | `q1E.invoke(q1, content)` |
| `q1.h(type)` | `q1H.invoke(q1, type)` |
| `q1.b()` | `q1B.invoke(q1)` |
| `send({type:'SEND-BUILD', line:...})` | `MonitorContext.logMessage("主动发送", ...)` |

Frida 脚本仍保留在 `frida/send/frida_active_send_rpc.js`，作为微信版本升级后的快速验证工具，不再是首选实现路径。

## 9. 已知限制

- 只覆盖文本类消息（`type=1`），其他类型（图片、视频、卡片、红包、转账等）不在本次范围。
- 仅适配微信 8.0.68 的混淆符号 `px0.r1` / `px0.q1`，以及主界面类名 `com.tencent.mm.ui.LauncherUI`。新版本如果重新混淆，需要同步更新 `sender.ActiveSender` 内的 `R1_CLASS` / `Q1_CLASS` 常量与 `HookTargets.LAUNCHER_UI`。
- 触发依赖用户切回微信主界面（LauncherUI.onResume），不会在后台自动轮询；如需后台触发请自行扩展为 HandlerThread 周期任务。
- 不对发送频率做限流，调用方自行控制以避免被微信风控。
- `SendResult.isOk()` 仅代表请求是否成功入队，不代表服务端是否签收；服务端结果体现在 `[发送回调] px0.r0.onGYNetEnd` 日志中。
