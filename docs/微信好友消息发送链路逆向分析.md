# 微信好友消息发送链路逆向分析

作者：花落

许可：MIT

## 1. 目标

本文记录微信 Android 8.0.68 中“给好友发送消息”的主链路，重点回答三个问题：

1. 文本消息最终走哪个 scene
2. 发送请求对象在哪一层组装
3. `talker`、`content`、`type` 等关键参数在哪一层落地

本次结论基于以下对象：

- `wx8068_mcp_decode/smali_classes9/px0/r0.smali`
- `wx8068_mcp_decode/smali_classes9/px0/q1.smali`
- `wx8068_mcp_decode/smali_classes9/px0/r1.smali`
- `wx8068_mcp_decode/smali_classes9/fo1/g.smali`
- `wx8068_mcp_decode/smali_classes9/com/tencent/mm/ui/chatting/gc.smali`
- `wx8068_mcp_decode/smali_classes9/com/tencent/mm/ui/chatting/component/ki.smali`
- `wx8068_mcp_decode/lib/arm64-v8a/libwechatnetwork.so`
- `WeChatMonitor/docs/libwechatnetwork_so消息发送链路逆向分析.md`

## 2. 总结结论

好友文本消息发送主链路如下：

```text
Chatting UI
  -> px0.r1.a(talker)
  -> px0.q1
     -> g(talker)
     -> e(content)
     -> h(type)
     -> b()
  -> fo1.g.j(q1)
  -> new px0.r0(...)
  -> px0.r0.doScene(...)
  -> /cgi-bin/micromsg-bin/newsendmsg
```

其中：

- `px0.q1` 是发送请求对象
- `fo1.g` 是把 `q1` 转成实际网络 scene 的工厂
- `px0.r0` 是真正的发送 scene
- `px0.r0` 对应日志名 `MicroMsg.NetSceneSendMsg`
- `px0.r0.getType() == 0x20a`
- `px0.r0.doScene()` 使用的 CGI 是 `/cgi-bin/micromsg-bin/newsendmsg`
- `px0.r0` 的旧 cmd id 是 `0xed`，与 `libwechatnetwork.so` functionlist 里的 `newsendmsg reqid=237` 对齐

## 3. Native 层定位

在 `libwechatnetwork.so` 中已经确认：

- 存在 `/cgi-bin/micromsg-bin/newsendmsg`
- 存在 `/cgi-bin/micromsg-bin/sendmsg`
- 存在 `reqid="237"` 对应 `newsendmsg`
- 存在 `reqid="2"` 对应 `sendmsg`
- 存在 `LongLinkTaskManager`、`ShortLinkWithMMTLS`、`MMTLS`、`mars::stn::Task` 等 mars 网络栈符号和日志

这说明普通消息发送能力最终落在网络任务分发层，但 native 层本身主要负责：

- `Task` 调度
- `Req2Buf`
- `StartTask`
- `OnTaskEnd`
- 长链/短链选择
- MMTLS 加密传输
- CGI functionlist 路由

而不是在 native 层直接组装聊天文本内容。

也就是说：

```text
Java 上层组装消息
  -> StnManager.Task
  -> libwechatnetwork.so 负责网络发送
```

更完整的 native 边界见：

`docs/libwechatnetwork_so消息发送链路逆向分析.md`

## 4. 发送 Scene：px0.r0

文件：`smali_classes9/px0/r0.smali`

类声明：

```smali
.class public Lpx0/r0;
.super Lcom/tencent/mm/modelbase/m1;
.implements Lcom/tencent/mm/network/l0;
```

从日志字符串可以直接看出它的用途：

```smali
const-string v0, "MicroMsg.NetSceneSendMsg"
```

### 4.1 doScene

`doScene()` 中明确写入：

```smali
const-string v2, "/cgi-bin/micromsg-bin/newsendmsg"
iput-object v2, v1, Lcom/tencent/mm/modelbase/l;->c:Ljava/lang/String;

const/16 v2, 0x20a
iput v2, v1, Lcom/tencent/mm/modelbase/l;->d:I

const/16 v2, 0xed
iput v2, v1, Lcom/tencent/mm/modelbase/l;->e:I
```

结论：

- 请求路径：`/cgi-bin/micromsg-bin/newsendmsg`
- scene type：`0x20a`
- 对应旧协议号：`0xed`

### 4.2 getType

`getType()` 直接返回：

```smali
const/16 v0, 0x20a
return v0
```

所以 `px0.r0` 就是 scene `522`。

### 4.3 构造函数参数

`fo1.g.j()` 调用的主构造函数是：

```smali
invoke-direct/range {v3 .. v8}, Lpx0/r0;-><init>(Ljava/lang/String;Ljava/lang/String;IILjava/lang/Object;)V
```

参数顺序为：

1. `String talker`
2. `String content`
3. `int type`
4. `int flag`
5. `Object extra`

### 4.4 本地落库

在构造函数内部，`px0.r0` 会先构造 `com.tencent.mm.storage.f8`，然后写入：

- `talker`
- `content`
- `type`
- `msgSource`

并先插入到消息数据库，再走网络发送。

这意味着：

```text
发送动作不是“直接发网络包”
而是“先建本地消息对象，再发网络 scene”
```

## 5. 请求对象：px0.q1

文件：`smali_classes9/px0/q1.smali`

`px0.q1` 是发送请求对象，重要字段如下：

| 字段 | 类型 | 当前确认含义 |
|---|---|---|
| `b` | `String` | `talker` |
| `d` | `String` | `content` |
| `e` | `int` | 消息 `type` |
| `f` | `int` | 发送附加标志 |
| `g` | `long` | 重发场景使用的本地消息 id |
| `h` | `Object` | 附加参数对象 |
| `i` | `int` | 发送模式 |
| `j` | `xv0.f7` | 扩展元数据 |
| `k` | `px0.o1` | 发送策略 |
| `n` | `String` | 额外字符串参数 |

### 5.1 builder 方法

已确认的 builder 方法：

```smali
g(String) -> iput-object p1, p0, Lpx0/q1;->b:Ljava/lang/String;
e(String) -> iput-object p1, p0, Lpx0/q1;->d:Ljava/lang/String;
h(I)      -> iput p1, p0, Lpx0/q1;->e:I
```

对应含义：

- `g()` 设置 `talker`
- `e()` 设置 `content`
- `h()` 设置消息类型

### 5.2 执行入口

`q1.b()` 最终调用：

```smali
invoke-static {p0, v0, v1, v0}, Lpx0/q1;->d(Lpx0/q1;Lqp5/l;ILjava/lang/Object;)V
```

继续进入：

```smali
invoke-virtual {p0, p1}, Lpx0/q1;->c(Lqp5/l;)V
```

因此 `q1.b()` 就是“发消息”的执行入口。

## 6. 工厂：px0.r1

文件：`smali_classes9/px0/r1.smali`

`px0.r1` 是发送请求对象的简化工厂。

### 6.1 创建 q1

`a(String)` 最终调用：

```smali
new-instance p2, Lpx0/q1;
invoke-direct {p2}, Lpx0/q1;-><init>()V
invoke-virtual {p2, p0}, Lpx0/q1;->g(Ljava/lang/String;)Lpx0/q1;
iput-object p1, p2, Lpx0/q1;->k:Lpx0/o1;
```

结论：

```text
px0.r1.a(talker)
  = 创建新的 q1
  = 先把 talker 写入 q1.b
  = 设置发送策略 q1.k
```

## 7. Scene 工厂：fo1.g

文件：`smali_classes9/fo1/g.smali`

`fo1.g.j(Lpx0/q1;)Lpx0/l1;` 是把 `q1` 转成真实网络 scene 的关键方法。

### 7.1 i 字段决定分支

它先看：

```smali
iget v1, p1, Lpx0/q1;->i:I
```

然后按 `i` 选择不同构造函数：

- `i == 5`：`new px0.r0(String, String, int, int, Object)`
- `i == 4`：`new px0.r0(String, String, int, int, long)`
- `i == 2`：`new px0.r0(long, int, String)`
- 其他：空 `r0`

### 7.2 i == 5 是当前主发送分支

关键代码：

```smali
iget-object v4, p1, Lpx0/q1;->b:Ljava/lang/String;
iget-object v5, p1, Lpx0/q1;->d:Ljava/lang/String;
iget v6, p1, Lpx0/q1;->e:I
iget v7, p1, Lpx0/q1;->f:I
iget-object v8, p1, Lpx0/q1;->h:Ljava/lang/Object;
invoke-direct/range {v3 .. v8}, Lpx0/r0;-><init>(Ljava/lang/String;Ljava/lang/String;IILjava/lang/Object;)V
```

这一步彻底把 `q1` 的字段映射到 `NetSceneSendMsg`。

## 8. 聊天界面入口

### 8.1 `com.tencent.mm.ui.chatting.gc`

文件：`smali_classes9/com/tencent/mm/ui/chatting/gc.smali`

关键片段：

```smali
invoke-static {p1}, Lpx0/r1;->a(Ljava/lang/String;)Lpx0/q1;
move-result-object v1

invoke-virtual {v1, p1}, Lpx0/q1;->g(Ljava/lang/String;)Lpx0/q1;
invoke-virtual {v1, p2}, Lpx0/q1;->e(Ljava/lang/String;)Lpx0/q1;
invoke-virtual {v1, p2_type}, Lpx0/q1;->h(I)Lpx0/q1;
invoke-virtual {v1}, Lpx0/q1;->b()V
```

这是最直接的文本发送链路：

```text
username -> q1.g(username)
content  -> q1.e(content)
type     -> q1.h(type)
execute  -> q1.b()
```

这里 type 根据场景取：

- `0x42`
- `0x2a`

### 8.2 `com.tencent.mm.ui.chatting.component.ki`

文件：`smali_classes9/com/tencent/mm/ui/chatting/component/ki.smali`

关键片段：

```smali
invoke-static {v6}, Lpx0/r1;->a(Ljava/lang/String;)Lpx0/q1;
move-result-object v7

iput v5, v7, Lpx0/q1;->e:I
iput v10, v7, Lpx0/q1;->f:I
invoke-virtual {v7, v11}, Lpx0/q1;->e(Ljava/lang/String;)Lpx0/q1;
invoke-virtual {v7, v6}, Lpx0/q1;->g(Ljava/lang/String;)Lpx0/q1;
iput-object v9, v7, Lpx0/q1;->h:Ljava/lang/Object;
iput-object v8, v7, Lpx0/q1;->n:Ljava/lang/String;
iput v5, v7, Lpx0/q1;->i:I
```

然后继续：

```smali
invoke-virtual {v7}, Lpx0/q1;->a()Lpx0/m1;
```

这说明 `ki` 是更完整的发送入口，除了基本的 `talker/content/type` 外，还会填：

- `f`
- `h`
- `n`
- `i`

其中 `i = 5` 会走到 `fo1.g.j()` 的主发送分支。

## 9. 完整发送链路

当前已经确认的“好友文本消息发送”主链如下：

```text
com.tencent.mm.ui.chatting.gc / component.ki
  -> px0.r1.a(talker)
  -> px0.q1
     -> g(talker)
     -> e(content)
     -> h(type)
     -> set f / h / n / i
     -> b() or a()
  -> fo1.g.j(q1)
  -> new px0.r0(...)
  -> px0.r0.doScene()
  -> /cgi-bin/micromsg-bin/newsendmsg
  -> libwechatnetwork.so / mars STN
     -> reqid=237
     -> longlink/shortlink/MMTLS
  -> onGYNetEnd()
  -> SendMsgSuccessEvent / SendMsgFailEvent
```

## 10. 成功与失败事件

`px0.r0.onGYNetEnd()` 中会根据返回值发布事件：

- `SendMsgSuccessEvent`
- `SendMsgFailEvent`

因此如果目标是监听“发送成功/失败”，除了 Hook `gc` 或 `ki` 的发起点，也可以直接 Hook：

- `px0.r0.onGYNetEnd(...)`
- `SendMsgSuccessEvent`
- `SendMsgFailEvent`

## 11. 逆向建议

### 11.1 如果要”监听发送”

优先 Hook：

- `com.tencent.mm.ui.chatting.gc`
- `com.tencent.mm.ui.chatting.component.ki`
- `px0.r0.onGYNetEnd`

已集成到 Xposed 模块（`WeChatMonitorHook.java`）的 `hookSendRequestChain()`：

| Hook 点 | 日志类型 | 内容 |
|---|---|---|
| `px0.r1.a(String)` | 发送链路 | talker |
| `px0.q1.g(String)` | 发送链路 | talker 写入 |
| `px0.q1.e(String)` | 发送链路 | content 写入 |
| `px0.q1.h(int)` | 发送链路 | type 写入 |
| `px0.q1.b()` / `px0.q1.a()` | 发送请求 | talker + content + type + mode |
| `px0.r0.onGYNetEnd(...)` | 发送成功/失败 | errType + errCode |

### 11.2 如果要“伪造发送”

最短路径是构造：

```text
px0.r1.a(talker)
  -> q1.e(content)
  -> q1.h(type)
  -> q1.b()
```

但是否还需要额外填：

- `q1.f`
- `q1.h`
- `q1.i`
- `q1.n`

取决于具体消息类型和当前聊天场景。

### 11.3 如果只发普通文本

当前链路显示最小必要字段是：

- `talker`
- `content`
- `type`

其余字段在部分调用点会补充，但文本消息主链已经可以通过 `gc` 看清。

## 12. Frida 动态验证脚本

```text
frida/frida_friend_send_chain.js
```

用途：被动 Hook 当前文档确认的 Java 发送链路，不修改参数和返回值，只打印发送请求构造、scene 创建、提交和回调。

覆盖点：

| Hook 点 | 作用 |
|---|---|
| `px0.r1.a(String)` | 观察 `talker -> q1` 创建 |
| `px0.q1.g(String)` | 观察 talker 写入 |
| `px0.q1.e(String)` | 观察 content 写入 |
| `px0.q1.h(int)` | 观察消息 type 写入 |
| `px0.q1.a()` / `px0.q1.b()` | 观察发送执行入口 |
| `fo1.g.j(px0.q1)` | 观察 `q1 -> px0.r0` scene 映射 |
| `px0.r0.<init>(...)` | 观察 `NetSceneSendMsg` 构造参数 |
| `px0.r0.doScene(...)` | 观察 scene 提交 |
| `px0.r0.getType()` | 验证 scene type `0x20a / 522` |
| `px0.r0.onGYNetEnd(...)` | 观察发送成功/失败回调 |

运行示例：

```bash
frida -U -f com.tencent.mm -l frida/frida_friend_send_chain.js
```


## 13. 主动发送 RPC 脚本

新增脚本：

```text
frida/frida_active_send_rpc.js
```

用途：不在注入时自动发送，只有手动调用 RPC 才触发发送。

调用方式：

```bash
frida -U -p 4106 -l "F:/nx/wx Android/WeChatMonitor/frida/frida_active_send_rpc.js"
```

进入 Frida 控制台后手动执行：

```js
rpc.exports.sendtext("wxid_kzo8vdr4d6zd22", "hello from frida", 1)
```

限制：

- 仅支持文本消息 `type=1`
- 必须手动调用，不会在脚本加载时自动发送
- 只适合你自己的授权测试账号

内部流程：

```text
px0.r1.a(talker)
  -> px0.q1.e(content)
  -> px0.q1.h(1)
  -> px0.q1.b()
```

## 14. 当前结论

本次已经确认：

1. 微信好友消息发送不需要先从 native 层找明文组包逻辑。
2. Java 层已经明确存在 `NetSceneSendMsg` 的完整构造链。
3. 真正的发送 scene 是 `px0.r0`。
4. 真实 CGI 是 `/cgi-bin/micromsg-bin/newsendmsg`。
5. `px0.q1` 是发送请求对象，`fo1.g` 是发送 scene 工厂。
6. 聊天界面直接通过 `gc` 和 `ki` 两条路径进入该链路。
7. `libwechatnetwork.so` 确认承载 `newsendmsg reqid=237` 的 mars 网络发送能力，但不是普通文本参数组装首选位置。

也就是说：

```text
如果目标是“找到如何给好友发送信息”
当前已经找到可操作的 Java 主链
不需要继续把重点放在 libwechatnetwork.so 的 native 组包细节上
```
