# 微信消息明文获取指南

## 概述

微信消息的解密发生在 **native 层**，但解密结果直接写入 **Java 消息对象字段**。因此获取明文**不需要 Hook native 层**，直接从 Java 层拦截消息对象的 getter 方法即可。

普通文本消息和支付消息走不同的回调链路，需要分别 Hook。

---

## 一、普通文本消息

### Hook 类与方法

| 类名 | 方法 | 返回值 | 说明 |
|------|------|--------|------|
| `ok.y7` | `j()` | `String` | 消息正文，无参 getter |
| `ok.y7` | `getContent()` | `String` | 消息正文，同上 |
| `com.tencent.mm.storage.f8` | `j()` | `String` | 存储层消息正文 |
| `com.tencent.mm.storage.f8` | `getContent()` | `String` | 存储层消息正文 |

`ok.y7` 和 `f8` 返回的是同一个消息内容，只是调用时机不同（一个是消息对象层，一个是存储层），实际抓取时会有重复，需要去重。

### 可获取的元数据字段

同一消息对象（`this`）还可获取：

| 方法 | 返回值 | 说明 |
|------|--------|------|
| `P0()` | `String` | 会话 ID（talker/对方 wxid） |
| `getTalker()` | `String` | 同上 |
| `getType()` | `int` | 消息类型（1=文本） |
| `getMsgId()` | `long` | 本地消息 ID |
| `getCreateTime()` | `long` | 创建时间戳 |

### 去重策略

`ok.y7` 和 `f8` 的输出内容完全相同，使用以下指纹去重：

```
fingerprint = "明文消息|" + talker + "|" + msgId + "|" + content
时间窗口 = 1500ms
```

同一指纹在 1.5 秒内只输出一次。

### 中文乱码处理

终端直接打印中文可能乱码，日志输出 `decodedUnicode` 和 `contentB64` 两个字段。`frida/frida_plaintext_probe.js` 会在脚本内对 `contentB64` 做 Base64 解码，再用 Unicode escape 输出，当前输出格式为：

```text
send_time_ms | talker | decodedUnicode=<Base64 decoded Unicode escape> | contentB64=<Base64(UTF-8)>
```

```text
1715358055000 | wxid_xxx | decodedUnicode=测试明文 | contentB64=5rWL6K+V5piO5paH
```

`decodedUnicode` 是脚本内 Base64 解码后的内容，只是用 ASCII 安全的 `\\u` 形式展示，避免终端乱码。


### Frida 实现

```javascript
// hookClassMethod 同时 hook 所有 overload
function hookClassMethod(className, methodName, tag) {
    const Cls = Java.use(className);
    Cls[methodName].overloads.forEach(function(ov) {
        ov.implementation = function() {
            const ret = ov.apply(this, arguments);
            logText(tag, className + '.' + methodName + ' =>', ret);
            return ret;
        };
    });
}

Java.perform(function() {
    hookClassMethod('ok.y7', 'j', 'MSG-GETTER');
    hookClassMethod('ok.y7', 'getContent', 'MSG-GETTER');
    hookClassMethod('com.tencent.mm.storage.f8', 'j', 'MSG-GETTER');
    hookClassMethod('com.tencent.mm.storage.f8', 'getContent', 'MSG-GETTER');
});
```

### Xposed 实现

```java
// XposedHelpers.findClass + findAndHookMethod
private void hookPlaintextGetterMethods(Class<?> msgClass, String className, String methodName) {
    for (Method method : msgClass.getDeclaredMethods()) {
        if (!methodName.equals(method.getName())
            || method.getParameterTypes().length != 0
            || method.getReturnType() != String.class) {
            continue;
        }
        XposedBridge.hookMethod(method, new XC_MethodHook() {
            @Override
            protected void afterHookedMethod(MethodHookParam param) {
                String content = (String) param.getResult();
                // 取元数据
                String talker = extractSimpleMethodValue(param.thisObject, "P0");
                String msgId = extractSimpleMethodValue(param.thisObject, "getMsgId");
                // 记录日志
                log("明文消息", "talker=" + talker + " msgId=" + msgId + " content=" + content);
            }
        });
    }
}
```

## 二、支付消息

支付消息走 `sysmsg` XML 格式，通过独立的监听器回调，不走普通文本消息链路。

### 数据格式

普通消息 type=1（文本），支付消息 type=49（appmsg），content 字段是 XML：

```xml
<sysmsg type="paymsg">
  <paymsg>
    <PayMsgType>16</PayMsgType>
    <WalletType>1</WalletType>
    ...
  </paymsg>
</sysmsg>
```

### Hook 方式

**不需要单独的支付回调类**。支付消息（type=49 appmsg）走的是和普通消息完全相同的管道：

```
xv0.wc.j(p0) → h8.U8(f8) → f8.j() / ok.y7.j()
```

在 `ok.y7.j()` / `f8.j()` 的返回值中，普通文本直接是明文，支付消息 content 是 `<sysmsg>` XML。通过检查 `type=49` + `content.contains("<sysmsg")` 即可区分。

### 解析链路

```
ok.y7.j() / f8.j() → String content (sysmsg XML)
  → aa.d(xml, "sysmsg", null)       ← 解析 XML → Map<String,String>
    → .sysmsg.paymsg.PayMsgType      ← 支付消息类型码
    → .sysmsg.paymsg.WalletType      ← 钱包类型
```

### 实现

`WeChatMonitorHook.emitPlaintextMessage()` 中内置支付检测：

```java
if ("49".equals(type) && content.contains("<sysmsg")) {
    String paySummary = parsePaymentXml(content);  // 调用 aa.d() 解析
    log("支付消息", details);
    return;
}
// 否则按普通明文消息处理
log("明文消息", details);
```

### PayMsgType 枚举

| 值 | 含义 |
|----|------|
| 0x10 (16) | 钱包类型改变 |
| 0x11 (17) | 钱包类型更新 |
| 0x25 (37) | C2C 内容更新 |

### XML 解析工具类

```java
// 微信内置 XML→Map 工具
Map<String, String> map = (Map<String, String>) XposedHelpers.callStaticMethod(
    XposedHelpers.findClass("com.tencent.mm.sdk.platformtools.aa", classLoader),
    "d",
    xmlString,    // XML 字符串
    "sysmsg",     // 根节点名
    null          // 前缀
);
// 取值示例
String payMsgType = map.get(".sysmsg.paymsg.PayMsgType");
String walletType = map.get(".sysmsg.paymsg.WalletType");
```

### 当前实现文件

- `WeChatMonitorHook.emitPlaintextMessage()` — 内置支付检测和 XML 解析
- `WeChatMonitorHook.parsePaymentXml()` — XML 解析 + 字段提取
- `PayMessageMonitor.java` — 独立的支付监听器（`oe0.n1` 不存在于 8.0.68，保留备用）

## 三、Native 层补充

| 目标 | 所在层 | 需要的 so |
|------|--------|-----------|
| 普通消息正文 | Java | 无需 |
| 支付消息类型 | Java | 无需 |
| 离线支付消息分发 | Native | `libkinda_android.so` |
| 收款消息监听回调 | Native | `libkinda_android.so` |
| 香港钱包离线通知 | Native | `libkinda_android.so` |
| 加密/解密算法 | Native | `libwechatmm.so` / `libMMProtocalJni.so` |
| 签名校验逻辑 | Native | `libtenpay_utils.so` |

**Java 层获取明文是最简单的方式。Native 层 `libkinda_android.so` 包含完整的离线支付消息分发链路，可用于验证和高级场景。详见 `libkinda_android_so逆向分析.md`。**

## 五、补充：发送链路

本文聚焦“接收侧明文”和“支付消息识别”。

如果目标是“找微信如何给好友发送消息”，当前主链已经单独整理在：

`docs/微信好友消息发送链路逆向分析.md`

发送侧目前已确认：

```text
Chatting UI
  -> px0.r1.a(talker)
  -> px0.q1
     -> g(talker)
     -> e(content)
     -> h(type)
     -> b() / a()
  -> fo1.g.j(q1)
  -> px0.r0 (MicroMsg.NetSceneSendMsg)
  -> /cgi-bin/micromsg-bin/newsendmsg
```

因此：

- 接收明文：优先看 `ok.y7` / `f8`
- 发送文本：优先看 `px0.q1` / `fo1.g` / `px0.r0`
- 支付 native 分发：优先看 `libkinda_android.so`
